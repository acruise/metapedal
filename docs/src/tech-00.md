## The Standard Module: Electronics

The standard motherboard is a microcontroller-based design. The choice of microcontroller class is one of the most important decisions in the specification, and it deserves serious thought because it constrains essentially everything else about the product experience.

There is a tempting option at the high end of the microcontroller spectrum, which is to use a part capable of running embedded Linux. The STM32MP series from STMicroelectronics is one example, with several other vendors offering similar Cortex-A class parts. The appeal of this approach is rapid prototyping. With Linux on the device, the development experience becomes essentially identical to working on a small computer: package managers, networking stacks, the ability to log in remotely and edit files, all the affordances that make iteration fast. For a one-person hardware project this is genuinely seductive.

The problem is that every one of those affordances becomes a liability in the deployed product. Linux has a userland and a kernel and a boot process and a filesystem that can corrupt and services that can hang. The boot time is multiple seconds at best. The attack surface is enormous. The thermal and power budgets are larger than they need to be. And most damningly, the customer mental model breaks: a guitar pedal that takes eight seconds to boot and occasionally needs a filesystem check is not really a guitar pedal anymore, it is a small computer pretending to be a pedal. The whole virtue of a pedal is that it is a fixed-function appliance with appliance-grade reliability.

The deeper technical issue with Linux is that it is not a real-time operating system. The kernel itself does work at unpredictable times, the scheduler is not designed for hard real-time, and even with real-time kernel patches there are still many places where unpredictable delays can occur. Low-latency audio requires deterministic execution, and Linux fundamentally cannot guarantee it. So Linux-class silicon is the wrong answer for the deployed product, even though it would be a nicer development experience.

## 9V Pedalboard Power Support

The platform's adoption story benefits significantly from being compatible with the existing pedalboard infrastructure. That infrastructure is genuinely trivial: 9-volt center-negative DC power on a standard barrel connector, and quarter-inch mono instrument cables for audio. Both standards are nearly a century old and universally compatible across pedal manufacturers. A user with an existing pedalboard already has the power supply and cables needed to integrate any compliant pedal into their chain.

Every Metapedal module accepts standard 9V center-negative pedalboard power alongside the USB-C power option. The 9V barrel jack on the back panel is the standard 2.1mm or 2.5mm DC barrel that all pedalboard power supplies use, with the polarity convention matching the established pedal standard. A small DC-DC converter on the motherboard steps the 9V input down to the various internal rails the module needs (typically 5V and 3.3V). The converter cost is a dollar or two in components for the buck regulator chip, the inductor, capacitors, and the barrel jack itself, which is trivial compared to the adoption-barrier reduction it provides.

The dual power input (9V barrel for pedalboard use, USB-C for chain-mode and host-connected use) is managed by simple input selection: whichever input is providing power is the one being used, with appropriate diode-OR or active switching to handle the case where both inputs are connected simultaneously. When both are present, the USB-C input typically takes precedence for the inter-module bus context, but the module continues to function the same way regardless of which power source is active.

This integration means a user can drop a single Metapedal module into their existing pedalboard with zero friction, using their existing power supply and existing quarter-inch cables. The module behaves like any other pedal in the chain, with its function determined by whatever preset is loaded. The Metapedal-specific features (inter-module bus, routing graph, chain-wide presets) only matter if the user has multiple Metapedal modules; with a single module, the user just gets a configurable digital effects pedal. This is the same low-friction adoption path that helped ExpressLRS grow within the existing RC transmitter ecosystem.

The 9V power input also handles the portable operation case nicely. The battery backpack accessory we discussed earlier provides 9V output through a barrel jack as one of its options, alongside the USB-C output. Users who want to power their pedalboard from a portable battery can use the 9V output to feed the standard pedalboard power supply input rather than needing to convert to USB-C. This makes the battery backpack a more versatile accessory that works with both Metapedal modules and any other pedalboard gear the user owns.

There is a subtle consequence for the power rail handling in chained modules. When modules are chained together over the inter-module bus, USB Power Delivery negotiation handles the power budget for the chain. If individual modules are also powered through their 9V barrel jacks from a pedalboard supply, the power management has to handle the case gracefully. The simplest approach is to have each module's power input feed its own internal regulators, with the inter-module bus carrying only data, not power, when modules are individually powered through their 9V inputs. The chain still operates correctly because the bus protocol works regardless of how each module is powered. Users who want a single power source can rely on USB-C power propagation through the chain; users who want pedalboard integration can power each module individually from the pedalboard supply.

## The A Motherboard: Functional Blocks and Modular Composition

A useful exercise at this point is to enumerate what is actually on the A motherboard, organized by functional block, with reasonable component selections and rough costs. This is not yet a literal manufacturable BOM but it is detailed enough to verify that the architecture is buildable and to establish a realistic cost target.

The organizing principle for the physical layout is connector-on-mezzanine, where every external connector lives on a small daughter board rather than directly on the motherboard. The rationale is mechanical stress isolation: any connector that protrudes from the enclosure will take stress from cables being yanked, stomps that bend things, and drops onto the connector edge. A connector mounted directly on the motherboard transfers that stress through its pins into the motherboard's pads and traces, which is exactly where you do not want it because broken pads on the motherboard mean the whole module is dead and probably unrepairable. A connector mounted on a small mezzanine takes the stress on the mezzanine, which is a cheap replaceable board. The pattern is common in laptops and phones where each external port typically lives on a small flex-cable-attached subboard that can be replaced independently.

For the A motherboard, the natural mezzanine groupings are: a back panel connector mezzanine carrying the 9V barrel jack, the two USB-C inter-module ports, the microSD slot, and the TRRS jack; an M8 peripheral mezzanine carrying the M8 Smart Digital port (this could be merged with the back panel mezzanine but keeping it separate isolates the M8 connector's mechanical stress to its own small board); a top panel mezzanine carrying the footswitch, the OLED display, the RGB LED, and the side-mounted navigation buttons, attached via flexible flat cable since the top of the enclosure needs to open for service. The A variant has no side panel mezzanines because it has no quarter-inch jacks.

The mezzanines connect to the motherboard through small board-to-board connectors (Hirose DF40 or Molex SlimStack family, 20 to 40 pins, 0.4 to 0.5 mm pitch, perhaps a dollar or two for the connector pair) or through flexible flat cables for the top mezzanine. The board-to-board connectors live inside the enclosure where they are not exposed to external forces.

The motherboard's functional blocks:

The processor block is at the center of the board. STM32H743 or similar STM32H7 family part, in a TQFP-176 or BGA-240 package, around six to nine dollars. Decoupling capacitors at the power pins (100 nanofarad and 4.7 microfarad ceramics, perhaps fifteen of them across the chip). The 25 megahertz HSE crystal sits within a few millimeters of the chip's HSE pins with its load capacitors. The exposed thermal pad has a via array to a ground plane. The processor block also includes boot mode selection circuitry, a small jumper or test point for DFU mode, and a 4 or 10 pin SWD debug connector exposed inside the enclosure for development access.

The power supply block handles conversion from the various inputs to internal rails. The inputs are 9 volts from the barrel jack and 5 volts from USB-C after USB-PD negotiation. The outputs are 3.3 volts digital for the logic, 3.3 volts analog (kept separate from digital 3.3 volts for noise reasons), and 5 volts available for any USB-host operation. The power architecture uses a buck converter from 9 volts or 5 volts down to 3.3 volts digital (a TI TPS62xx or similar, around eighty cents), an LDO from 3.3 volts digital to 3.3 volts analog (giving noise rejection between the rails, perhaps thirty cents), and a power-path selector that picks between the 9 volt and USB-C inputs through two ideal-diode-OR controllers like the LTC4357 (a dollar or two total). USB-PD negotiation happens through a dedicated PD controller chip on the USB-C port (TI TPS65987 or CY7C65215 family, two to three dollars), which negotiates with upstream ports for appropriate voltage and current.

The USB-C inter-module communication block handles the two USB-C ports, both running at high-speed (480 Mbps) to give the bandwidth headroom that the platform's actual use cases demand. The H7 cannot natively drive two high-speed USB ports because its OTG FS controller is full-speed only and only its OTG HS controller can reach high-speed (and that requires an external ULPI PHY chip). So the architecture uses external silicon to deliver symmetric high-speed operation on both user-facing USB-C ports.

The architecture: an external USB 2.0 hub chip (Microchip USB2422 or similar dual-port high-speed hub, $1.50 in production quantities) connects to the H7's OTG HS controller through one upstream port and provides two high-speed downstream ports to the user-facing USB-C connectors. The OTG HS controller in turn connects to the hub through an external ULPI PHY chip (Microchip USB3300 or TI TUSB1210, $1 in production quantities) that provides the physical layer for high-speed operation. The total BOM addition for the high-speed dual-port architecture is $2-3, paid by every module.

This is the right choice despite the BOM premium. A chain with eight to twelve modules (which the platform should support) carries more total bandwidth at full speed than full-speed can comfortably handle. A B2 module operating as a USB audio interface needs to match the Scarlett 2i2's 2-in/2-out 24-bit/96 kHz capability credibly, which at full speed leaves no headroom for control traffic and USB overhead. Multi-track MIDI playback alongside live processing pushes bandwidth use further. Version two's multi-performer chains and feature expansions add more bandwidth needs that full-speed cannot absorb. Committing to high speed in version one means the platform has architectural room to grow without restructuring later.

The H7's OTG FS controller becomes available for other uses since the inter-module bus does not need it. It could drive the external display output through a separate USB-C port, or could be used for some future expansion that needs a third USB connection. Useful flexibility for future capabilities even if version one does not use it.

The hub chip becomes a single point of failure for both USB ports. If the hub fails, both ports stop working. This is mitigated by hub chips being mature commodity parts with high reliability, and by the platform's modular architecture making module replacement cheap. The dual-controller-no-hub architecture would have been slightly more robust against single-chip failure, but the cleanliness and symmetry of the hub-everything-through-one-chip architecture wins on balance.

The PD controllers manage power and CC line behavior on each port (TI TPS65987 or CY7C65215 family, $2-3 total for both ports). The USB 2.0 data lines run from each USB-C connector through the back-panel mezzanine connector to the motherboard, where they meet the hub chip's downstream ports. High-speed USB at 480 Mbps requires careful PCB layout with 90-ohm differential impedance, controlled-length matching of the differential pair, ground references under the differential pair across its full length, minimal vias, and careful return path management. The H7's reference designs from STMicroelectronics include high-speed PHY examples that establish the layout patterns. ESD protection at the connectors uses TVS arrays rated for high-speed USB rather than generic USB (parts like the TI TPD4S012 are specifically designed for this and remain in the same cost range). CC lines carry orientation detection and PD communication. VBUS lines have inrush limiting and bulk capacitance.

The audio output block has the audio-grade DAC chip driving the TRRS jack's stereo output. The DAC choice is something like the TI PCM5102 (about a dollar, stereo 24-bit, 192 kilohertz, I-squared-S input). The DAC takes I-squared-S audio from the STM32H7's SAI peripheral and produces analog stereo at line level. The DAC output feeds an op-amp-based headphone amplifier (TI TPA6132A or similar, forty cents, driving 32 ohm headphones at reasonable levels), which feeds the TRRS jack's tip and ring contacts. Output coupling capacitors (47 microfarad electrolytics, two of them) provide DC blocking. Output protection (small resistors and TVS) handles shorts at the jack.

The audio input block for the A variant is much simpler: a basic microphone preamp feeding the STM32H7's built-in 16-bit ADC. A single op-amp stage with around 30 to 40 decibels of gain (an OPA1612 if you want quality, or a cheaper part like the LMV358 if you do not), with an input coupling capacitor and bias network for the electret microphone in typical TRRS headphones. Anti-aliasing filter (RC plus perhaps one op-amp stage) before the ADC. The whole block is maybe four passives and one op-amp, totaling well under a dollar.

The H7's internal ADC is more capable than its baseline specification suggests. The hardware is a 16-bit successive approximation converter with additional oversampling, with noise-limited typical performance around 14-bit equivalent (about 86 decibels SNR) at the native 5 MSPS sampling rate. Through oversampling and decimation, the effective resolution can climb higher, with ST documenting up to 21-bit effective resolution under appropriate conditions. For the A variant's audio-to-control extraction use case, 14-bit baseline performance is comfortable, and additional resolution through oversampling is available if the firmware wants it. The ADC is not the limiting factor on what the A variant can do.

What the A variant cannot do, and what makes the B mezzanine worth its cost, is handle the electrical chaos of real instrument and microphone signals. This is worth being explicit about because the bit-count comparison between the H7's ADC and a codec chip is the wrong way to think about the A versus B distinction. The H7's ADC specification is a clean-lab number, assuming a well-conditioned signal arriving at the input with appropriate amplitude, appropriate impedance matching, no DC offset, no common-mode noise, no ground-loop contamination, and a clean ground reference. Real instrument signals violate every one of those assumptions.

A guitar plugged into a quarter-inch jack is connecting through a long unbalanced cable that picks up RF interference from radio stations, fluorescent lights, dimmer switches, switching power supplies, and computer monitors. The pickup itself is a coil of wire that acts as an antenna for any nearby magnetic field, particularly the 60 hertz hum from power transformers. The signal level ranges from millivolts (clean fingerpicked passages on a humbucker) to volts peak-to-peak (aggressive strumming on a high-output pickup), and the input stage has to handle this dynamic range without clipping or losing quiet signals in noise. The source impedance varies wildly across pickup types, from around 5 kilohms for a vintage single-coil to near zero for active pickups, and the input stage's response has to remain musically appropriate across all of them. Ground loops between the instrument's amplifier, the pedalboard's power supply, the venue's electrical system, and whatever the user is touching show up as 60 hertz hum and various harmonics modulated by everything from refrigerator compressors cycling to dimmers being adjusted.

A microphone signal is even harder. Dynamic microphones produce millivolt-level signals that have to be amplified by 40 to 60 decibels before they reach line level. The preamp doing that amplification has to keep its own noise floor below the level of the amplified signal, which requires careful low-noise op-amp selection, careful power supply design, careful PCB layout to keep digital noise away from the analog path, and careful grounding to prevent amplified ground noise from drowning the signal. Phantom power adds 48 volts of DC that has to be supplied cleanly through the same connector that carries the signal, with appropriate decoupling to keep the phantom supply from contaminating the audio path. None of this is hard in principle, but all of it requires real analog engineering attention.

The B mezzanine's audio codec chips solve all of this internally. A part like the Cirrus Logic CS4272, the Texas Instruments PCM3060, or the AKM AK4954 includes not just an ADC but a complete analog front-end: differential input stages that reject common-mode noise, programmable gain amplifiers that handle dynamic range, anti-aliasing filters that handle out-of-band signals, and internal voltage references that are isolated from the digital domain. The codec has been designed and tested specifically for audio applications, with the analog gotchas already worked out. The user gets professional audio quality by buying the chip rather than by engineering a discrete preamp around the H7's ADC.

The cost difference between the two approaches reflects this. The codec chip is two to four dollars in production quantities, but it replaces the discrete preamp components that would otherwise be needed. The total cost of "H7 ADC plus careful discrete preamp" approaches the cost of "codec chip" once you add up the precision op-amps, matched resistor networks, low-noise voltage references, careful PCB layout work, and the engineering effort to make it all work reliably across manufacturing variations. The codec is not just better quality; it is also cheaper in real engineering terms because the analog work is amortized across the codec manufacturer's volume.

The A-versus-B distinction is therefore fundamentally about analog engineering investment, not about raw ADC bit count. The A variant's microphone input is appropriately matched to its use case: a clean TRRS-jack electret microphone signal that does not have to handle instrument-level dynamic range, instrument-level source impedance variation, or instrument-level electrical chaos. The H7's ADC with minimal analog conditioning works fine for that use case, and the audio-to-control extraction infrastructure built on top of it does useful work. The B variant is the configuration that handles real instrument signals, real microphone signals, real ground loops, and real electrical chaos, and it earns its codec chip by being engineered specifically for those conditions. The B mezzanine is not a luxury upgrade over the A variant; it is a fundamentally different product for a fundamentally different signal environment.

The OLED display block has I-squared-C lines from the STM32H7 going through the FFC to the top mezzanine, where they connect to a self-contained 1.3 inch SSD1306-based OLED module (two to three dollars in volume). The motherboard just provides power and I-squared-C; the OLED handles everything else internally.

The user input block has the footswitch (a quality 3PDT or DPDT mechanical switch, two to three dollars), four side-mounted tactile buttons (ten cents each, forty cents total), and an RGB LED for the top panel indicator (either a single addressable LED like a WS2812 for twenty cents, or a discrete RGB LED with three current-limiting resistors for ten cents). All connect through the top mezzanine's FFC to motherboard GPIOs.

The motion sensing block uses a 6-axis IMU (inertial measurement unit) combining a 3-axis accelerometer and a 3-axis gyroscope. Parts like the Bosch BMI270, the TDK InvenSense ICM-42688, or the ST LSM6DSV cost around $1.50 to $3 in production quantities, with I-squared-C interface, similar package size, and similar power consumption to accelerometer-only parts. The IMU lives on the motherboard since the motion it measures is the same regardless of where on the board it sits.

The IMU is a meaningful upgrade over a pure accelerometer for several reasons. The basic orientation-detection use case (pedalboard versus hand-held configuration mode) benefits because the gyroscope tracks rotation independently of gravity, handling the transition states cleanly. With accelerometer-only, picking up a module from the pedalboard creates motion that confuses the orientation detection during the transition. With the IMU, sensor fusion (typically a Mahony or Madgwick filter, running in a few hundred clock cycles per sample on the H7) produces robust orientation estimates even during fast movement, and the transition from pedalboard to hand-held position is detected immediately and cleanly.

More interestingly, the IMU becomes a music control source through gesture recognition. A tilt-controlled wah effect maps the module's pitch axis to wah frequency, with rocking the module forward and back modulating the effect. A whammy-style pitch shift maps the roll axis to pitch. A shake-triggered tremolo extracts the shake frequency and uses it as a tempo source. A free-form modulation source assigns IMU outputs to arbitrary parameters in the routing graph. The module's motion becomes an expressive control surface, comparable in capability to a dedicated expression pedal but built into every module.

The hand-held configuration mode also gets richer with gesture recognition: rotating the module clockwise to advance through menu items, rotating counterclockwise to go back, tilting forward to select, tilting back to cancel, shaking to reset. The whole menu navigation interface can be gesture-driven without requiring the user to find and press the small side buttons. The user holds the module in their hand and steers through the menus by moving the module itself.

The footswitch stomp also becomes a richer event with the IMU watching. The footswitch provides a clean digital signal when stomped, but the IMU sees the same stomp as an acceleration spike with magnitude information. A drummer using a Metapedal module on the floor to trigger samples could get velocity-sensitive triggering from the stomp force, with harder stomps producing louder samples or triggering different sample variations.

Tap tempo through case taps is another natural feature that falls out of the IMU. The user taps the side of the case with their fingertip while playing, and each tap registers as an acceleration spike that the firmware identifies through a simple peak detector with a refractory period. After three or four consecutive taps, the firmware extracts the tempo from the intervals between taps and updates the chain master tempo accordingly. Time-based effects (delays, tremolos, choruses) sync to the new tempo automatically. The interaction is natural for performers who are already holding their instrument: their hands are busy with the guitar, their foot is on the footswitch, but the side of a fingertip or the back of a knuckle can still tap the metal case without breaking playing flow. The tap tempo capability is not specific to the module that has the tap input: the user might tap any module in the chain, with the resulting tempo event flowing through the routing graph to wherever it has been directed.

A few interesting variations are available within the same mechanism. Tapping a rhythmic pattern rather than just steady quarter notes lets the firmware extract meter information from the relative intervals, not just tempo. The firmware can distinguish taps at different case locations by the IMU's axis signatures, so top taps, side taps, and back taps can be treated as separate event sources if the user configures them that way. And the tap detection naturally extends to other surfaces the module sits on: a tap on the pedalboard near the module shows up as a smaller but distinguishable acceleration event, useful if the user wants to set tempo from a drumstick tap or another performer's foot.

The default behavior should probably be that tap-the-case is disabled unless explicitly enabled, since accidental tempo changes from setting the module down or bumping it would be more disruptive than the inconvenience of having to enable the feature first. The user opts in through a configuration setting per-module or per-preset, and from that point the tap-tempo capability is active. The footswitch can also serve as a mode toggle: a long press on the footswitch puts the module into tap-tempo mode for the next few seconds, with the IMU listening for taps and exiting the mode after the tempo is established.

The same tap-detection infrastructure extends naturally to drumming on the module case as a performance input. Rather than just extracting tempo from regular taps, the firmware can capture the full rhythmic pattern as a stream of timestamped events with velocity information from the impact magnitudes. A drummer working out a rhythm pattern can tap it out on the side of a module, and the system captures the pattern as a MIDI file on the microSD card for refinement in a DAW, or routes the trigger events through the routing graph to drive samples or synthesized drum sounds in real time. The Metapedal becomes a small portable drum pad that fits in a bag. A percussionist on stage can tap the side of a module as a performance instrument, with the IMU's velocity sensitivity giving expressive control over volume and tone. Multiple modules in a chain become a multi-pad surface where tapping different modules produces different drum sounds because the routing graph maps each module's tap events to different samples.

The IMU can distinguish between different kinds of taps to some extent: a sharp finger tap, a flat-hand slap, a knuckle rap, and a fingernail click all have different acceleration signatures. The firmware can be configured to map different tap types to different drum sounds, multiplying the effective polyphony of a single module's tap surface without requiring more modules. This is sophisticated firmware work but the underlying detection is well within what the IMU and the H7 can handle. The drumming-on-modules feature shares its enable flag with tap tempo: both are off by default to prevent accidental triggers from bumps and setting the module down, and the user enables them when they want the IMU to behave as an input rather than just an orientation sensor.

The IMU's vibration rejection is also useful in performance contexts. On a stage with subwoofers thumping or with the user dancing around their pedalboard, a pure accelerometer might trigger spurious orientation changes from sustained vibration. The IMU's sensor fusion correctly identifies vibration as oscillation around a stable orientation rather than as actual orientation change, and the false triggers go away. This matters more than it might seem because failures of orientation detection during live performance would be both annoying and embarrassing.

The microSD card block has the card slot (push-push or hinged-cover, fifty cents) on the back mezzanine, connected to the STM32H7's SDMMC peripheral. SDMMC peripheral supports high-speed modes natively. The connection is six signal lines (clock, command, four data lines) plus power and ground, all running through the back mezzanine's board-to-board connector. ESD protection on each signal line at the connector.

The TRRS jack block has the TRRS connector itself (a 3.5mm four-conductor jack with side or back mount, around a dollar) on the back mezzanine. Signal routing through the mezzanine connector to the motherboard: audio output block drives tip and ring (left and right), audio input block reads from the second ring (microphone), sleeve is audio ground reference. The MIDI signal selection circuitry (an analog switch like the TS3A24159 that routes the same physical contacts to MIDI signals when configured for MIDI mode rather than audio) lives on the back mezzanine near the jack.

The M8 peripheral block has the M8 Smart Digital connector on its own small mezzanine, with I-squared-C or SPI signals routing through the connector to the STM32H7. ESD protection on each signal line. Power output to the peripheral (3.3 volts, current-limited through a load switch like the TPS22918, perhaps fifty cents) that the firmware can enable or disable.

The pin header block is the internal expansion pad array for hacker modifications. A 20 or 40 pin 0.1 inch pitch pad array on the motherboard, exposing GPIOs, SPI, I-squared-C, UART, and various power rails. The pads are unpopulated by default; users install their own header, solder wires directly, or install through-hole components as their project requires. Sits inside the enclosure, accessible only when the enclosure is opened.

The 9V barrel jack block has the barrel jack (thirty cents) on the back mezzanine, with the 9V rail routing through the mezzanine connector to the motherboard's power-path selector. Reverse polarity protection (a P-channel MOSFET in the high side, twenty cents) sits near the jack on the mezzanine to catch the case where a user plugs in a wrong-polarity supply.

The ground and power planes use the four-layer stackup we already specified: signal layers on the outside, a continuous ground plane plus a power plane in the middle. Digital ground and analog ground are both on the ground plane, separated only by component placement rather than by a physical plane gap (the modern preference for continuous ground planes with careful layout). The power plane carries 3.3 volts digital and 3.3 volts analog as separate regions, with a tie point near the audio DAC. The 5 volt and 9 volt rails are distributed on the outer signal layers as power traces rather than plane regions, since they only feed specific blocks.

Every component listed is a commodity part available from multiple suppliers. No exotic silicon, no custom mechanical parts, no specialty processes. A small contract manufacturer can produce this in volumes of hundreds without difficulty, and at volumes of thousands the per-unit cost drops further as components move from low-quantity pricing to true production pricing.

The connector-mezzanine pattern adds maybe three to five dollars to the total cost compared to mounting all connectors directly on the motherboard, but the repairability and stress-isolation benefits are substantial. A stomped connector becomes a five to ten dollar repair of swapping out a mezzanine board, rather than a forty to fifty dollar replacement of the whole module. For a product intended for performance contexts where rough handling is normal, this is genuinely valuable.

## The Mezzanine Architecture

The Metapedal motherboard provides core compute and communication; mezzanine boards provide specialized I/O. A mezzanine attaches to the motherboard through a 50-pin board-to-board connector and delivers whatever the mezzanine designer chooses: an audio codec with quarter-inch or combo jacks for B variants, a fat DSP chip for BX, CV/gate interfaces for modular synth integration, or specialty I/O for niche applications.

The platform ships the following standard mezzanines in v1: B1 (one combo jack), B2 (two combo jacks), BX (B1 or B2 plus an Analog Devices SHARC fat DSP for users wanting neural amp modeling, large convolution reverbs, or other heavy algorithms), B16 (sixteen-channel multitrack), B32 (thirty-two-channel multitrack). The A variant has no mezzanine; its simple audio components live directly on the motherboard.

A particularly important capability of every B variant is that, when connected to a host computer, it functions as a class-compliant USB audio interface. The B1 is functionally an entry-level audio interface comparable to a Focusrite Scarlett Solo or PreSonus AudioBox iOne. The B2 is functionally a dual-input audio interface comparable to a Focusrite Scarlett 2i2. The same modules are also useful as standalone effect processors when not connected to a computer, which means they serve as two products in one: USB audio interface plus standalone DSP unit. This dual use is enabled by the existing USB host capability and the USB Audio Class 2.0 implementation that the platform specification has already committed to.

Beyond the standard mezzanines, the platform publishes the mezzanine bus specification as an open contract that lets third parties build conforming mezzanines. The bus specification covers the 50-pin connector pinout, the lease-based I2S resource protocol, the identification protocol, multi-mezzanine configurations through splitter mezzanines, mechanical constraints, power distribution, and compliance testing. The full specification is in the dedicated mezzanine bus document; this technical document captures only the high-level architectural commitments and the v1 product specifics.

The key architectural commitments that affect the rest of this document:

The 50-pin board-to-board connector with a specific pinout is the v1 hardware commitment. Four I2S data lines (SDI, SDO, SDI2, SDO2) plus two independent clock domains (BCK_A/WS_A/MCLK_A and BCK_B/WS_B/MCLK_B) form the audio interface; I2C and SPI buses provide control; analog I/O, GPIO, and power rails round out the connection. Mezzanines claim subsets of the four I2S data lines through a lease protocol that the motherboard arbitrates.

The mezzanine grounding architecture uses the board-to-board connector's geometry to achieve effective analog/digital ground separation. The motherboard's digital ground plane and the mezzanine's analog ground plane meet at the connector with multiple ground pins providing low-inductance connection while the physical separation provides inherent isolation between digital noise sources and sensitive analog circuitry. This is a meaningful design property that the platform's architecture inherits naturally from the mezzanine approach, with the dedicated mezzanine bus document covering the design considerations in detail.

The mezzanine is attached securely through the connector plus two M2 screws into threaded standoffs, with the platform's mechanical specification ensuring reliable retention through stage abuse, transport, and repeated mating cycles. Detailed mechanical specifications are in the dedicated mezzanine bus document.

For complete details on the mezzanine bus specification, including pin allocation, lease protocol, identification protocol, power distribution, mechanical specifications, compliance testing, and guidance for third-party mezzanine designers, refer to the dedicated Metapedal Mezzanine Bus Specification document.

## Storage: Internal Nonvolatile Memory and the microSD Slot

The platform has two distinct storage needs with very different characteristics. Parameter and preset storage is small in size but needs to be reliable across power cycles and accessible quickly during operation. Audio recording, sample library storage, and firmware update images are large in size and benefit from removable media that users can swap and access from a computer.

The internal nonvolatile storage on the H7 handles the first need. The H7 family has 4 kilobytes of battery-backed BKPSRAM that survives power-down if an external coin cell is connected. This is useful for very small persistent state like the last-active preset number or critical calibration data, but is too small for general parameter storage. The internal Flash memory, ranging from 128 kilobytes to 2 megabytes depending on the specific H7 variant, is usable as EEPROM emulation for parameter storage. Flash has limitations as a data store: the erase granularity is large at typically 128 kilobytes per sector, the write endurance is limited to roughly 10000 cycles, and writes stall code execution from the same Flash bank. But for the kinds of state Metapedal needs to persist (preset banks, routing graph configurations, calibration data, user nicknames), the limitations are manageable with appropriate wear-leveling. Realistic preset usage even with aggressive editing stays well within the endurance limit, and the rare-write nature of preset data does not stress the code-execution stall issue.

So the internal Flash handles parameter storage adequately without requiring any external nonvolatile memory. The standard motherboard does not need an external EEPROM chip or other dedicated parameter storage; the H7's internal resources are sufficient.

The microSD card slot handles the bulk storage need. A microSD slot on every B variant module gives the platform local storage for audio recording, sample libraries, firmware update images, and any other high-volume content. The H7 has a dedicated SDMMC peripheral that handles the SD card protocol natively, with DMA support so audio recording does not steal CPU cycles from DSP processing. The interface is well-supported by existing open-source libraries; the FatFs library by ChaN is the canonical filesystem implementation and runs well on the H7.

The bandwidth requirements for audio recording are modest. A stereo 24-bit 48-kilohertz recording is about 280 kilobytes per second. A B2 with both channels recording at full quality stays within this rate. Modern microSD cards handle this bandwidth with significant headroom; even older Class 10 cards sustain 10 megabytes per second of write throughput, which is more than thirty times the actual requirement. The user can record continuously without worrying about the storage falling behind the audio stream.

Storage capacity is similarly favorable. A 32-gigabyte microSD card holds about 30 hours of stereo 24/48 audio, and cards in this capacity range cost a few dollars retail. Users can essentially have unlimited recording time for any realistic session. Higher-capacity cards extend this further: a 128-gigabyte card holds well over 100 hours of stereo recording.

The recording format defaults to uncompressed WAV, which is universally compatible with audio software and is straightforward to write in real time. FLAC is available as an option for users who prioritize smaller file sizes at the cost of additional CPU work during recording. Lossy compression formats like AAC or Opus are deliberately not offered because they introduce audio quality concerns that recording users typically want to avoid. The file format choice is a preset parameter that the user sets per recording.

The filesystem is FAT32 or exFAT depending on card size, both of which are universally readable on computers. Users pull the SD card out of the module and read the recordings as standard files on macOS, Windows, or Linux. No special software is required to access the recordings; they show up as ordinary audio files in any standard file browser. This is a critical user experience requirement because the alternative (proprietary file formats requiring vendor software to read) is the kind of vendor lock-in that the platform's open philosophy specifically avoids.

The slot's physical location matters for user experience. It belongs on a panel that is accessible without disassembling the module, probably the back panel where the connectors live. A push-push mechanism gives positive tactile feedback when the card is fully seated and ejects the card when pressed again, which is the right user interaction for media that gets inserted and removed regularly. The connector itself costs roughly fifty cents to a dollar in production quantities and occupies modest board space. The PCB cost increase from the slot itself plus the higher-speed SDMMC signal routing is small.

What the SD card enables is worth being explicit about because the storage capability multiplies what the platform can do.

Audio recording is the most obvious use case. The user assigns a footswitch or other control to start and stop recording. The module captures whichever signals the routing graph configures it to capture: dry input, processed output, both simultaneously on separate channels, or any other source available in the chain. The recording happens locally without requiring a connected computer. The user pulls the card out after the session and imports the files into their preferred audio software for editing and mixing.

The loop pedal use case is more sophisticated than typical commercial loop pedals because the SD card extends what is possible. A loop pedal captures a phrase, plays it back, and layers additional phrases on top. Commercial loop pedals typically store loops only in volatile memory, so power-cycling loses everything. A Metapedal-based loop pedal with SD storage saves loops between sessions, lets users build libraries of loops for later use, and combines loops from previous sessions with new ones to construct larger compositions. A street performer can develop a library of backing phrases at home and use them in different combinations during shows.

The multitrack recorder use case scales with the platform. A single B2 captures two channels to the SD card simultaneously. Adding additional B variant modules to the chain expands the channel count, with the inter-module bus aggregating audio streams to a single recording-coordinator module that handles SD card I/O. Four channels of simultaneous recording from two B2 modules plus a coordinator handles many singer-songwriter recording sessions or small ensemble capture without requiring a laptop.

The sampling use case turns the module into a sampler instrument. The user records short audio clips to the SD card during setup, then plays them back as triggered samples during performance. MIDI events or footswitches trigger sample playback. The sample library lives on the SD card, so a single small card can hold the user's complete sample collection portably. Loading new sample libraries is as simple as swapping cards.

The MIDI file playback use case is a particularly nice extension that combines the storage capability with the synthesizer infrastructure. A user loads a MIDI file onto the microSD card and configures a module to play it back through the onboard synthesizer engine. MIDI files are small (a few kilobytes for a typical song), so the SD card holds essentially unlimited backing tracks and accompaniment material. The user has a portable instrument that can practice along with backing tracks, perform with pre-recorded accompaniment, or rehearse parts in isolation from the rest of a band.

The integration with the existing platform features is clean. The MIDI playback is a control source in the routing graph, producing MIDI events at the timestamps specified in the file. These events flow into the routing graph and trigger synthesizer voices the same way events from a USB MIDI controller would. The synthesizer does not know or care that the events came from a file rather than a live performer.

Multi-track MIDI files enable additional capabilities. A typical MIDI file has separate tracks for drums, bass, keys, melody, and so on. The user can selectively enable or disable tracks to play along with the song. A guitarist practicing a part can disable the guitar track and play their part live while the rest of the band plays automatically. The same MIDI file becomes a versatile practice tool. Different tracks can route to different synthesizer instances within a module or across modules in the chain: drums to a drum synthesizer, bass to a bass synthesis, piano to a Hammond emulation, with the routing graph handling the distribution.

Playback control is just another routing graph action. A footswitch starts and stops playback. Tempo and pitch can be modified at playback time: a backing track in B-flat can be played in C with a whole-tone shift, or slowed to half speed for practice. Standard MIDI files use a well-documented binary format that is straightforward to parse in C or Rust, with existing parsers readily available.

The firmware update use case uses the SD card as a delivery mechanism. The user downloads a firmware update from the project's website, copies it to the SD card, inserts the card into the module, and the module updates itself. This is more user-friendly than the USB DFU path for users who do not want to connect to a computer for updates, and it works in offline contexts where the user does not have a computer available.

The microSD slot is probably worth including on every B variant rather than treating it as a board option, because the recording use case is appealing across many user contexts and the cost is modest. The A variant probably does not need a slot, since the use cases that motivate it are mostly audio-related and the A variant is the controller-grade variant without primary audio I/O. Whether the A variant gets a slot anyway, for sample library storage or sequencer pattern storage in future controller products, is a possible future enhancement but not a version one requirement.
